На рабочем проекте уже есть процесс тестирования UI, поэтому расскажу как устроено у нас.
Мы пишем UI на Jetpack Compose. Есть своя дизайн-система и отдельная команда, которая создает core компоненты.
Не все из этих компонент подходят нам в виде as is, поэтому есть и своя библиотека компонент, реализованных либо с нуля, либо в виде обертки над core компонентами.
На все компоненты у нас есть Preview (возможность без сборки проекта смотреть, как выглядит UI). 
А полноценный UI как правило состоит из этих компонент со своей стилизацией. На эти экраны так же пишутся Preview функции.
И уже на Preview пишутся UI (Screenshot) тесты.

Для тестирования используется библиотека paparazzi, сам тест выглядит как unit-тест с той разницей, 
что тест наследуется от специального класса из paparazzi, в который передается список типов устройств (размеры, типы экрана), для которых нужно сгенерировать скриншоты.
И далее есть сущность screenshotTester, в лямбду которой передается Composable Preview функция.
Скриншот генерируется и хранится в git lfs.

# Выводы

Актуальность screenshot тестов для нас стала актуальной, когда в компании создали отдельную команду дизайн-системы.
И нужно было отслеживать, что с поднятием версий ничего не ломается. На первых порах часто ехала верстка из-за изменений в шрифтах, цветах или типографии.

Кроме этого наличие скриншотов помогает понять, что вообще за фича находится в модуле и пройти пользовательский путь.

Поднятие версий Compose так же затрагивало UI, особенно в части отступов и цветов.
Поэтому после примерно года затаскивания процесса тестирования UI могу заключить, что изменения в UI теперь делаются легче, а количество ui багов, доходящих до тестирования и пользователей стало минимальным.
